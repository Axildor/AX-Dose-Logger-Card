## Key Design Decisions
1. **Full parity with Daily Panel (user-confirmed scope)** — the Drinks Panel expandable mirrors the Daily Panel structure exactly: Log Drink button override grid (mirrors take_pill grid), In Body Box expandable (mirrors Safe to Take Box), Disruption Box expandable (mirrors Pills Left Box), and Custom Chips expandable (mirrors chips).
2. **3-option select for Disruption box instead of a boolean toggle (Option A, user-confirmed)** — the three display modes (Sleep Disruption / Low - Timestamp / Low - Hours Until) are mutually exclusive states of one box. A single select dropdown is cleaner than a boolean toggle + conditional sub-select. The built-in mode swap wins over `disruption_entity` (mirrors Pills Left Box `pills_left_show_days_left` winning over `pills_left_entity`).
3. **Disruption box default tap = Sleep Disruption popup (card-internal, only when mode='disruption' + substance)** — retains the current behavior as the fallback. For `low_timestamp` / `low_hours_until` modes, the default tap falls back to more-info (like Safe to Take). A custom `disruption_tap_action` always wins.
4. **In Body box mirrors Safe to Take box (not Pills Left box)** — the In Body box default tap is more-info (no card-internal dialog), exactly like the Safe to Take box. `_handleInBodyBoxAction` follows the `_handleSafeBoxAction` signature (no fallback parameter).
5. **Separate `drink_chip_*` namespace (not reusing `chip_*`)** — a card is bound to one device, but shared chip fields would carry over confusingly if the user changes `device_id` from medicine to master tracker. Separate fields keep the two panels chip configs fully independent.
6. **No backend changes** — all three sensors (`sleepDisruption`, `estimatedLowTime`, `lowHoursUntil`) are already resolved in `ResolvedEntities` by `_computeEntities`. The card already has access to all the data; this is purely a frontend config + render change.
7. **Separate controller methods, not a refactor** — `_handleInBodyBoxAction` and `_handleDisruptionBoxAction` are separate methods (mirrors `_handleSafeBoxAction` / `_handlePillsLeftBoxAction`), avoiding any regression risk on the working Daily panel action paths. `_handleDisruptionBoxAction` generalizes with an optional `fallback` parameter for the card-internal Sleep Disruption popup default.
8. **`_relevantStateChanged` watches drink chip entities always (parallel to the Daily chips)** — they may belong to other devices and only render on the drinks pane, but keeping them always watched mirrors the existing Daily chip pattern (simpler than a pane-conditional guard, and the cost is negligible — at most 4 extra entity-id string comparisons per state-change check).

## Key Design Decisions
1. **Gate on the active pane to avoid needless re-renders** — The granular drink entities (stock, add_stock, avg7, avg365) are only displayed on the Inventory pane. The `if (this._activePane === 'inventory' && entities.substance)` guard skips the `_getDrinksOfSubstance` call entirely on other panes, so Daily/Drinks/Graphs/Stats/Tools panes pay zero cost. This mirrors the existing `changedProperties.has('_tick')` pane-conditional pattern in `shouldUpdate`.
2. **Watch all four rendered entity types** — The Inventory panel's `_renderRow` reads `stockEntityId`, `addStockEntityId`, `avg7EntityId`, and `avg365EntityId`. Watching all four ensures any of them changing state (a refill increments the stock number; an avg sensor updates on a new dose) triggers an immediate re-render. Watching only `stockEntityId` would miss avg-sensor updates that also display on the pane.
3. **The backend was already correct** — `DrinkStockNumber.add_stock` increments `_attr_native_value` and calls `async_write_ha_state()` immediately on the `drink_add_stock_{entry_id}` signal. The stock entity's HA state updates instantly; the bug was purely that the frontend's render-gate didn't compare that entity's state.
4. **The 30s `_tick` timer was the fallback** — `_tick` is whitelisted for the inventory pane in `shouldUpdate`, which is why the value eventually refreshed (~30s delay). The fix provides immediate feedback instead of waiting for the tick.
5. **No regression on the Drinks pane** — Logging a drink updates the master coordinator, which pushes to Master Tracker sensors already in `ResolvedEntities` (body mass, sleep disruption, last dose, avg), so the Drinks pane already re-renders correctly. The Inventory pane was unique in depending on granular entities not in the master's `ResolvedEntities`.

## Key Design Decisions
1. **Compress the taller panel to match the reference (Stats), not grow the reference (user-confirmed 2026-07-06)** — The Stats panel is the reference target; the Inventory `.avg-cell`'s two stacked lines made it the taller box (~84px vs Stats ~72px). Compressing the `.avg-cell` metrics (padding/gap/value-size) to match the Stats `.stat-cell` natural height is the content-driven approach. The col-1 `.stat-pill` follows automatically via grid `align-items: stretch` — no `.stat-pill` CSS change is required.
2. **No shared explicit min-height** — A shared `min-height` was rejected: it is a floor, not a ceiling, so it would not compress the Inventory row when content overflows, and pairing it with `max-height` + overflow would risk clipping the two-line `.avg-cell`. Matching the natural content height by tuning padding/gap/value-size keeps the box self-sizing if a long translated label ever wraps.
3. **`.avg-value` 18px→16px is the one deviation from the prior Daily/Drinks visual-parity baseline** — The earlier Inventory parity pass sized `.avg-value` to 18px to match `.stat-value` for visual parity. The height-parity goal takes precedence for col-2's two-line cell; `.stat-label`/`.stat-value` (Daily/Drinks verbatim) and `.avg-label` (13px) stay unchanged.
4. **CSS-only change; render/TS logic untouched** — No backend / coordinator / store / localize / types / config-flow change. No entity, no config field, no translation affected. README NOT updated (visual tweak, not a user-facing feature/installation change). No `projectstructure.md` change.

### Prior decisions (Tools Panel Box Style Parity, 2026-07-06)

1. **Classify by a backend `role` STATE ATTRIBUTE, not `entity_id` suffix and not `unique_id`** — Two matching keys were tried and both fail: (a) `entity_id` suffix — HA derives `entity_id` from `slugify(translated_name)` via [`async_generate_entity_id`](../../usr/src/homeassistant/homeassistant/helpers/entity.py:114) + [`util.slugify`](../../usr/src/homeassistant/homeassistant/util/__init__.py:39), and every drink/master entity sets `_attr_has_entity_name = True`, so the entity_id is the slugified *name* (`DrinkStockNumber` → `number.coffee_inventory`, no `_drink_stock` suffix) — the old suffix match silently missed every entity except `DrinkLogButton` (coincidental `log_drink` slug). (b) `entityInfo.unique_id` — **NOT present on the frontend `hass.entities` map**: HA's `config/entity_registry/list_for_display` websocket ([`components/config/entity_registry.py`](../../usr/src/homeassistant/homeassistant/components/config/entity_registry.py)) populates `hass.entities` from `_as_display_dict` ([`entity_registry.py:265`](../../usr/src/homeassistant/homeassistant/helpers/entity_registry.py:265)), which emits only `ei`+`pl`+display fields, OMITTING `unique_id` (only `config/entity_registry/list`, used by the entity editor, includes it). The first fix's `unique_id` switch made everything worse (Log Drink popup unpressable, refill boxes non-clickable, all values still `-`). **State attributes ARE present on `hass.states[entityId].attributes`** (the full state object), integration-controlled, and survive renames — already proven by `device_type`/`substance`/`pk_model`. A dedicated `role` attribute is the HA-correct, robust classification key. The stored `DrinkInfo`/`ResolvedEntities` fields stay as `entity_id`s (panel calls `getState(entityId)`/`openMoreInfo(entityId)`); only the matching key changes.
2. **One resolver fix repairs five symptoms** — The `role`-based classification fixes the Inventory `-` (stock + 7-day avg + trailing-dynamic avg), unblocks `drinkDaysSinceReveal` (needs `avg365EntityId` to read `history_start_date`), repairs the Drinks-pane Disruption N/A + Last "Never" (master `_computeEntities` suffix matches were the same bug class), restores the master Amount-in-24h Stats row, AND silently repairs the Tools panel's Undo/Reset. No per-panel fork.
3. **Visual parity via verbatim Daily/Drinks CSS, with one per-element case override** — `.stat-pill`/`.stat-label`/`.stat-value` adopted the Daily/Drinks verbatim baseline (primary-tinted `rgba(rgb-primary, 0.06)` transparency, no border, `12px 14px`/`8px`, `20px` icon at `opacity: 0.7`, `18px/600` value). The 2-column `.inv-row` grid + 380px responsive fallback are preserved (no Daily equivalent — correct layout for a per-drink list). `.avg-cell` adopts the same primary-tinted surface so col-2 shares the Daily card surface.
4. **Natural-case drink-name label (user-confirmed)** — Per user decision (2026-07-06), the Inventory panel's `.stat-label` deliberately OMITS `text-transform: uppercase` even though Daily/Drinks apply it. Rationale: col-1's label is the drink's configured proper-noun name ("Coffee", "Espresso"), which should keep its natural case, unlike Daily's "SAFE TO TAKE"/"PILLS LEFT" generic labels. All other Daily/Drinks label styling (size, color, letter-spacing, weight) is inherited.
5. **Backend `role` attribute is additive, no migration** — The `role` attribute is set in `_attr_extra_state_attributes` on each entity; existing entries gain it on next HA restart. No entity-registry change, no unique_id change, no translation change, no stored-data-shape change. The backend unique_ids remain correct (they were never the bug); `role` just gives the frontend a stable, present-on-the-frontend classification key.
6. **README NOT updated** — A UI style tweak + a silent bug fix introduce no new user-facing feature, entity, config field, or installation step. Per the README update rule, this is not logged there.
7. **Document the failed unique_id attempt to prevent recurrence** — The first fix's `unique_id` switch is a tempting-but-wrong approach that "looks" like HA best practice (unique_id IS integration-owned/stable) but fails because the frontend entity-registry display dict omits it. The memory bank + plan explicitly record this so the same mistake isn't repeated on future classifier work.

## Key Design Decisions
1. **Root cause: different label rendering mechanisms** — Entity selectors (`ha-entity-picker`) render an external label above the control box; text selectors (`ha-textfield`) render an internal floating label inside the control box. In a CSS grid row, the entity picker's control box is pushed down by the external label height while the text field's control box starts at the top of the grid cell, causing vertical misalignment.
2. **CSS fix, not schema fix** — HA's `ha-form` does not expose a schema property to force external label rendering on text selectors. The `inset-label` property exists on `ha-input` but `ha-form` controls how labels are passed and doesn't expose it via the schema. The only viable fix without a fully custom editor is CSS alignment.
3. **`align-items: end`** — Forces all grid children to align by their bottom edges. The shorter text field sinks to the bottom of the grid row, matching the entity picker's bottom edge. This aligns the physical input boxes regardless of the label height difference above them.
4. **Shadow DOM injection via MutationObserver** — `ha-form` uses shadow DOM, so external CSS can't reach its internal grid divs. The `MutationObserver` catches `ha-form` elements as they appear (the config dialog opens lazily) and injects the style tag into each form's `shadowRoot`. The style tag is id-tagged (`ax-dose-grid-align-items-end`) so it's only injected once per shadow root.
5. **All field labels restored** — The previous approach of removing labels (returning `undefined` from `computeLabel`) was reverted because it stripped UX context. The CSS alignment fix preserves all labels while fixing the visual misalignment.
